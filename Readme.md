# Запуск
Войдите в Nix-оболочку:
```bash
nix develop
```
Запустите локальный демон MariaDB:
```bash
start-mdb
```
Запустите процесс генерации и экспорта сырых блоков таблиц:
```bash
run-bench
```
Погасите MariaDB после тестов:
```bash
stop-mdb
```

# Результат
Полученные .raw файлы — это точные копии пространства таблиц InnoDB (.ibd). При их анализе постдедуп алгоритмом необходимо учитывать:
- Размер блока — 16384 байта (16 КБ): Ваш скрипт чтения файлов для бенчмарка дедупликации должен читать файлы блоками по 16384 байт.
- Системные заголовки страниц: Каждая 16 КБ страница в InnoDB начинается с FIL Header (первые 38 байт), где хранится контрольная сумма страницы, ее тип и LSN (Log Sequence Number). Из-за динамического LSN и чексумм две логически одинаковые страницы с данными будут иметь разные первые 38 байт. Это отличная проверка для алгоритмов со скользящим окном (CDC), которые смогут перешагнуть этот заголовок и дедуплицировать тело страницы, в отличие от фиксированных 16 КБ блоков.
- Сценарий 5_binary (Раздувание строк / BLOB): Если размер строки или бинарного объекта (LONGBLOB) в InnoDB превышает примерно половину размера страницы (~8 КБ), MariaDB автоматически выносит избыточные данные на отдельные Overflow-страницы (Uncompressed BLOB pages). В основном файле они будут идти друг за другом в виде чистых 16 КБ блоков.
